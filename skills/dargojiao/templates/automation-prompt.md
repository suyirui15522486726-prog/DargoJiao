使用 `$dargojiao` 处理飞书群 `{{FEISHU_CHAT_NAME}}` 中的短视频面试分享链接，并把笔记写入知识库 `{{FEISHU_WIKI_URL}}`。

配置：

- 本地项目：`{{LOCAL_PROJECT_PATH}}`
- 只读 Git ref：`{{GIT_REF}}`
- 时区：`{{TIMEZONE}}`
- 回看窗口：最近 `{{LOOKBACK_DAYS}}` 天
- 计划：按用户配置的工作日执行

执行规则：

1. 读取并遵守 `$dargojiao`、`lark-shared`、`lark-im`、`lark-wiki` 和 `lark-doc`。先检查飞书用户授权、网络、群读取、Wiki 读取与 Docx 写入权限。
2. 预检失败时，在 Codex「已安排」任务结果中以“⚠️ 短视频面试笔记执行异常”开头，写明时间、阶段、简要原因和用户动作。失败不能推进处理状态，也不能写入成功标识。
3. 完整分页读取回看窗口内的新增用户消息，忽略机器人、系统消息和已撤回消息，提取链接及同条文本中的作者、标题和话题。
4. 使用 `svnote-msg:<message_id>`、规范化 URL 和平台视频 ID 做处理前去重。标题只作冲突提醒，不单独决定跳过。
5. 优先读取原链接；页面受限时，按精确作者和标题核验同一作品或可信镜像。证据不足时不猜测、不写笔记、不占序号、不写成功标识，记录为“待重试”。
6. 生成连续序号、标题、总结时间、面试问题 Topic、错误答题和高分答题。无法取得逐字证据时把错误回答标为“反例归纳”，不得冒充视频原话。
7. 只读检查 `{{LOCAL_PROJECT_PATH}}` 的 `{{GIT_REF}}`。仅在存在真实、可定位代码证据时添加“结合项目谈一谈”，并附相对路径、源码符号和带语言标记的代码块；不要切换分支或修改工作区。
8. 按 Topic 写入最合适的现有 Wiki 文档；没有合适分类时才创建顶层 Docx。写入前再次去重，写入后执行回读验证，确认主标识和编号标题各出现一次。
9. 单条失败不影响其余链接。始终在 Codex「已安排」任务结果中报告成功数、跳过数、待重试数、目标分类、异常阶段和用户动作。
10. 不在任何输出中包含 token、Cookie、真实资源 ID、个人本地路径或代理地址。不安装媒体下载、转码、语音识别、常驻服务或数据库组件，也不调用外部内容解析服务。
