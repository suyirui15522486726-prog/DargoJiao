# 权限边界

## 两层权限

飞书操作同时受两层控制：

1. OpenAPI 授权：应用和用户令牌是否具备消息、Wiki、Docx 等接口能力。
2. 资源成员权限：当前用户是否在目标群和知识空间中，是否能查看历史消息并编辑目标文档。

第一层通过不代表第二层通过。遇到“接口有权限但文档拒绝访问”时，优先检查知识空间成员权限。

## 最小能力

DargoJiao 只需要：

- 读取指定群的历史消息。
- 读取指定 Wiki 的节点结构。
- 读取并编辑目标 Docx。
- 必要时在该知识空间创建顶层 Docx。

不需要通讯录全量读取、审批、日历、邮箱、管理员或组织架构权限。推荐先使用 `lark-cli auth login --recommend`，再由实际缺失权限提示补充最小业务域。

## 身份

- 读取群消息、Wiki 和 Docx 时使用用户身份。
- 不把用户 token 复制到配置、提示词、环境示例或 GitHub Issue。
- 授权失效时重新登录，不通过浏览器 Cookie 或手工 token 绕过。

输入群、Wiki 和 Docx 必须对完成 `lark-cli auth login` 的同一个用户身份可见。浏览器里另一个飞书账号能打开文档，不代表 CLI 当前身份也有权限。

## 写入确认

首次配置只在用户指定的测试文档中做写入预检。正式运行只追加 DargoJiao 生成的笔记和去重标识，不删除、移动或重排无关内容。

当目标链接无法解析、知识空间成员权限不足或写入后无法回读时，不得换到其他文档继续写，也不得留下成功标识。
